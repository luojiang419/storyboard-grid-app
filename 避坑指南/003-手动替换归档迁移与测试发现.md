# 手动替换归档迁移与测试发现

## 问题一：界面合并来源不等于数据库真正统一来源

- 资源栏按 `StoryboardCutAsset.imageId` 分组，在刷新映射时临时覆盖 `imageId`，只能让界面看起来合并。
- 用户要求“所有结果共享稳定来源 ID”时，新记录和历史记录都必须落实到数据库。
- 迁移前应先创建稳定的 `imported_images` 记录，再在事务中同步更新 `cut_results.image_id` 与对应 `cut_tasks.image_id`，避免两张表来源不一致。
- 旧的导入图片记录可能还有未知引用，不应在没有完整引用审计时激进删除；保留孤立旧记录比误删共享数据安全。

## 问题二：数据库路径与运行时路径口径不同

- 工程数据库按设计保存 `generated_images/...` 相对路径，便于工程整体移动。
- 控制器和画板模型使用经过项目路径解析器还原的绝对路径。
- 迁移测试应分别验证：数据库字段为新的相对路径、控制器资源和画板引用为可访问的绝对路径。
- 不能直接断言数据库路径等于运行时绝对路径，否则会把正确实现误判为失败。

## 问题三：历史画板名称要按可靠性分级恢复

- 当前仍在画板中使用的替换资源，优先从画板条目反查画板名称。
- 旧结构为 `<画板 UUID>/manual_replacements`，未被使用的历史资源可从父目录 UUID 匹配画板。
- 两种方式都无法恢复时使用“历史画板”，不能根据原文件名猜测画板名称。
- Windows 非法字符必须复用统一文件名清理逻辑，保证目录名和文件名口径一致。

## 问题四：测试写了不代表 CI 会运行

- Flutter 默认只自动发现 `*_test.dart`。
- `storyboard_controller_batch_flutter_check.dart` 只有显式指定路径才会执行，普通 `flutter test` 不会发现。
- 通过 `storyboard_controller_test.dart` 入口调用批量检查的 `main()`，可让既有控制器测试全部进入默认测试和 GitHub Actions，无需复制测试实现。
- 修改测试入口后必须观察全量测试数量确实增加；本轮从 221 项增加到 286 项。
