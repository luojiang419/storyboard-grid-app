# 损坏 PNG 解码必须失败关闭

## 问题现象

手动替换图片测试中发现：文件扩展名虽然是 `.png`，但 PNG 数据损坏时，`package:image` 的 `decodeImage` 不一定返回 `null`，也可能直接抛出 `ImageException`（例如 `Invalid IDAT checksum`）。如果替换入口只判断扩展名或只处理 `null`，异常会穿透到界面并中断替换流程。

## 根因

- 扩展名只能用于初筛，不能证明文件内容是有效图片。
- 不同损坏方式的解码失败表现不同：可能返回 `null`，也可能抛出解码异常。
- 系统文件拖放允许用户把任意文件放到目标格，因此不能信任输入内容。

## 解决方式

1. 先按支持的扩展名初筛。
2. 在后台 Isolate 中真实解码图片并读取宽高。
3. 捕获图片解码异常，将结果统一转换为 `0 x 0`。
4. 控制器检测到无效尺寸时提示“所选文件不是有效图片”，不复制、不登记、不替换当前格。
5. 自动化测试同时覆盖有效图片和扩展名正确但内容损坏的文件。

## 后续约束

凡是新增图片导入、拖放、替换或批处理入口，都必须复用“扩展名初筛 + 内容解码校验”的失败关闭策略；不要把 `decodeImage == null` 当作唯一失败分支。
