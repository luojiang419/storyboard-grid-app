# 编组展开状态与拖放历史测试

## 问题一：编组内来源点击后立即收起

- 资源栏同时存在“未编组来源集合”和“全部来源集合”。
- 展开状态清理只能依据全部有效来源 ID，不能依据当前未编组集合。
- 如果在 `build` 中按未编组集合执行 `removeWhere`，编组内来源刚加入展开集合就会在同一次重建中被删除，表现为标题显示“xx 张裁切图”但无法展开。
- “展开全部”的判断与写入也必须使用同一份全部来源 ID，避免状态口径不一致。

## 问题二：AnimatedCrossFade 会让存在性断言产生假阳性

- `AnimatedCrossFade` 收起时仍可能保留第二子树，只是处于离屏/不可命中状态。
- Widget 测试不能只断言 `find.byKey(...)` 存在；验证展开功能时应使用 `finder.hitTestable()`，确认缩略图真实可交互。

## 问题三：拖放测试 API 与历史伪记录

- 当前 Flutter 测试 SDK 的 `dragFrom` 不接收 `duration` 命名参数；需要固定时长时使用 `timedDragFrom(start, offset, duration)`。
- 仅在统一 `_replaceBoard` 中比较最终画板快照还不够：某些无效操作可能先触发自适应布局归一化，从而看起来发生了变化。
- 对“移除不存在图片”这类语义无效操作，应在入口提前返回；同时在历史入口保留最终快照相等判断，形成双层保护。

## 问题四：标题栏新增按钮必须验证极限宽度

- 故事板中栏宽度会被左侧资源栏和右侧参数栏共同挤压，测试中最低可到约 150px。
- 在标题栏新增默认 `IconButton` 时，单枚按钮默认占用约 48px，连续增加按钮很容易造成 `RenderFlex overflow`。
- 标题栏高频图标按钮应统一使用紧凑的 32x32 约束，并保留 Tooltip；修改后必须同时跑常规宽度和“资源栏受限布局”测试。
- 仅缩小按钮仍可能在极窄中栏触发溢出。窄布局应让标题使用可裁切的剩余区域，并用 `Stack + Positioned` 把固定宽度操作区钉在右侧；宽布局再用 `Row` 展示完整元数据。
